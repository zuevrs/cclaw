# Skill: frontend-ui-engineering

UI work is where the most common cclaw regressions land: an agent autogenerates a "looks good" component that ships purple gradients, a `rounded-2xl` everywhere, oversized padding, centered-everything layout, no focus indicators, no semantic HTML, and a `div onClick` that screen readers cannot reach. The diff passes the type-checker and the test suite green; the user opens the page and the experience is broken in five ways nobody tested.

This skill is the canonical citation target for UI-stage findings. It pins **what good frontend looks like** in terms cclaw's reviewer can cite: component-architecture rules (composition / controlled-vs-uncontrolled), design-system adherence ("don't invent — use the system"), the AI-aesthetic anti-pattern table that catches the autogenerated default, the WCAG 2.1 AA baseline (focus / semantic / ARIA-only-when-needed), and responsive-design principles. Stage-windowed on `["build", "review"]` because UI surfaces only ride those stages; triggers include `touch-surface:ui` and any of the standard frontend file extensions.

## When to use

- **When implementing an AC with `touchSurface: ui`** — read this skill before drafting the component tree. The component-architecture rules govern composition; the design-system section governs visual choices; the accessibility checklist governs the `verifiable` line on the AC.
- **When the diff adds or modifies a `.tsx` / `.jsx` / `.vue` / `.svelte` / `.html` / `.css` file** — apply the checks below regardless of whether the AC mentioned UI. The file extension is the trigger; the slug doesn't need to be "a UI slug" for the rules to apply.
- **In fix-only** when the reviewer cited an "AI aesthetic" anti-pattern finding (severity `consider` or higher), a missing focus indicator (severity `required`), a semantic-HTML violation (severity `required`), or a responsive-design layout that breaks under 320px viewport.
- **Reviewer-side** when scoring a UI surface. Cite this skill body for: "is this following the design system, or inventing tokens?", "does this match the AI-aesthetic anti-pattern table?", "does keyboard navigation reach every interactive element?".

## When NOT to apply

- **The slug is inline / trivial and doesn't touch UI files.** No UI surface, no checks to run. The skill stays idle.
- **The project has no design system.** This skill's "use the system, don't invent" rule presupposes the project has tokens / a component library to defer to. On a green-field project where the slug is *establishing* the system, the design-system section is informational — read it before deciding which library to adopt, but the "don't invent" rule is suspended until the system exists.
- **You are writing a one-off internal tool with no end-user.** The accessibility baseline is mandatory for user-facing UI; internal tooling can document the deviation and ship without it. The deviation goes in the slim summary's `Notes:` line; "internal tool" is not a shortcut, it is a logged exception.
- **The "AI aesthetic" you are about to ship is actually the project's pinned brand.** Some design systems legitimately use a heavy purple-gradient hero or oversized padding — that's a brand decision, not an anti-pattern. The anti-pattern table is the default warning; the project's design-system tokens win whenever they pin a value.
- **The user explicitly asked for an experimental layout** that breaks one of the rules. Record the deviation in the AC's `Notes:` and surface as a slim-summary line; do not silently ignore the rule.

## Component architecture

Two iron rules that govern composition.

### Rule 1 — Composition over configuration

A component that takes seven boolean props is two components. A component that takes a `variant` enum of more than three values is probably the wrong abstraction. A component whose JSX renders five different shapes depending on props is hiding a slot pattern.

**Heuristic:** if your prop list has > 4 fields, or > 2 booleans, ask "can this be composition?". The composition shape is:

```tsx
<Card>
  <CardHeader>
    <CardTitle>Stripe webhook</CardTitle>
    <CardActions><Button>Retry</Button></CardActions>
  </CardHeader>
  <CardBody>{events.map((e) => <EventRow key={e.id} event={e} />)}</CardBody>
</Card>
```

— not:

```tsx
<Card
  title="Stripe webhook"
  showActions
  actionLabel="Retry"
  bodyKind="event-list"
  body={events}
  ...
/>
```

The composition shape pushes choices to the call-site (where the context lives) instead of pre-deciding everything inside the component (where the context is gone). The configuration shape grows boolean by boolean until nobody can read it.

### Rule 2 — Controlled vs uncontrolled, pick one per component

Every form / interactive component carries internal state about value, focus, hover, expanded, disabled, error. The choice is **always** either:

- **Controlled** — the parent owns the state, passes it in as `value` + `onChange`. The component renders deterministically from props.
- **Uncontrolled** — the component owns the state internally (via `useState` / `ref`). The parent reads results via `onSubmit` / `defaultValue` / `ref.current`.

A component that lets the parent set `value` **and** keeps internal state is "partially controlled" and is a bug-magnet — the two sources of truth drift, and the test suite cannot tell which is canonical. Pick one mode; document which in the component's JSDoc.

Default to **controlled** for forms (the parent's state is the spec). Default to **uncontrolled** for transient UI (a modal's open/closed state can live inside the modal if nothing else cares). When in doubt, **controlled** — it is more verbose but every state transition is auditable.

## Design-system adherence

The rule is "use the system, don't invent tokens". Three concrete cases:

| situation | what to do | what not to do |
| --- | --- | --- |
| The design system pins a token (e.g. `space.md = 16px`) | use `space.md` / `padding="md"` / `class="p-4"` (whichever the framework expects) | hard-code `padding: 16px` |
| The design system has no token for the value you need (e.g. an irregular gap) | propose adding the token (PR to the system) or revisit the design — irregular values almost always mean the design drifted from the system | invent a one-off token inline, hard-code the value, "this one is special" |
| The design system has a component for the surface (e.g. `Button`) | use the component | re-implement a button from a `<div onClick>` ("but I needed a custom shape") |

Three rules that fall out of the above:

1. **Tokens are the contract.** Colours, spacing, radii, typography, shadows, motion — every value is a token. Hard-coded values bypass the system; they survive light-mode but die on dark-mode / brand-refresh / RTL.
2. **Components are the contract.** Don't `<div onClick>` what `<Button>` already does. The system's `Button` carries focus / hover / disabled / loading / aria-pressed states; your `<div>` carries none. The reviewer cites unsystem buttons as `accessibility` with `required` severity.
3. **Variants live in the system, not in app code.** If your app needs a `<Button variant="destructive">`, propose adding the variant to the system. Do not extend the component locally with `className="bg-red-500"`. Local extension is design-system entropy.

When the design system is missing or doesn't cover the surface (a one-off marketing page, a prototype, the first slug of a new project), document the deviation in the slim summary's `Notes:` and the AC's frontmatter (`design_system: ad-hoc`). The reviewer treats `ad-hoc` as a `consider` finding (worth flagging) rather than a `required` one (broken).

## AI-aesthetic anti-pattern table

The autogenerated default. Catch yourself reaching for these; do the right column instead.

| AI aesthetic (avoid) | what to do instead |
| --- | --- |
| **Purple-to-pink gradient hero** (the "AI demo" look) | Use the design system's accent — usually a single brand colour, on white or a 5-8% tint background. Gradients are a deliberate brand choice; do not default to them. |
| **`rounded-2xl` everywhere** | Match the design system's radius scale. Most systems use 2-3 radius tokens (e.g. `radius.sm`, `radius.md`, `radius.lg`); `2xl` is a flag the agent reached for a Tailwind preset without thinking. |
| **Oversized padding** (`p-12` / `p-16` on small components) | Match the system's spacing scale. A button does not need 48px of internal padding; a card does not need 64px around the content. Generous whitespace is a design choice; overgeneration is the AI default. |
| **Center-everything** (`flex items-center justify-center` on every container) | Left-align text by default. Center is for marketing heroes, full-screen empty states, and modal pickers. Centered body text and forms are a tell. |
| **Drop-shadow on everything** (`shadow-xl` on cards, buttons, inputs) | Most surfaces should have no shadow or a single ambient shadow level. The system pins shadow tokens (`shadow.sm`, `shadow.md`); stop layering them. |
| **Soft-white-on-soft-white** (cards on background that have insufficient contrast) | Cards need a visible boundary: a 1px border, a 4% darker background, or a subtle shadow — pick one. Three-layer soft-white is invisible on most monitors. |
| **Lucide-icon-on-every-line** (decorative icons in lists where text is enough) | Icons are functional, not decorative. Use one when it adds information (`<CheckIcon> Completed`) or affordance (`<ChevronDownIcon>` for expand). Do not sprinkle them on every list row. |
| **Animations on every state** (`transition-all duration-300` on every component) | Animate state changes that need affordance (focus / hover / open / closed). Do not animate `padding`, `border-radius`, `gap` mid-render. CLS punishes the page; the user notices. |
| **Generic placeholder text** ("Enter your text here..." / "Type something...") | Placeholders describe the expected value's *format*, not its semantic name. The label is the semantic name. "Email" is the label; "name@company.com" is a useful placeholder; "Enter email" is redundant. |
| **Sentence-case button labels** ("Click here" / "Save changes" / "Continue to next step") | Match the system's button-label convention (usually title-case or imperative-verb-noun). Long button labels are flags the action is unclear; rename. |

The table is a list of **defaults to avoid**, not absolute prohibitions. Every entry's right-column rule is overridden by an explicit design-system pin or a brand decision; surface deviations in the slim summary's `Notes:` line.

## WCAG 2.1 AA accessibility baseline

The minimum accessibility bar for any user-facing UI. Failing any of these is a `required` reviewer finding.

### Focus and keyboard

- Every interactive element has a **visible focus indicator** (a 2px outline / ring at minimum, with ≥ 3:1 contrast against background). Remove `outline: none` only when replacing it with an equivalent or better indicator.
- The tab order matches the visual order. Skip-links are present at page top when the layout is complex (a "Skip to main content" link).
- Modal dialogs **trap focus**: tab cycles inside the open dialog; escape closes it. The trigger receives focus back on close.
- Custom interactive elements (e.g. a `<div role="button">`) handle both `Enter` and `Space` keys. The system's `<Button>` does this for you; that is one reason to use it.

### Semantic HTML

- Headings nest (`h1` → `h2` → `h3`); do not skip levels (`h1` directly to `h4` is a structure bug).
- Lists use `<ul>` / `<ol>` / `<li>`, not `<div>`.
- Buttons use `<button>`. Links use `<a>` with `href`. The distinction is semantic: links navigate, buttons act. A `<button>` that navigates is a bug; an `<a>` that mutates state is a bug.
- Form controls have associated `<label>` elements (`<label for="id">`, or wrapping). Placeholders are **not** labels.
- Tables use `<table>` / `<thead>` / `<tbody>` / `<th scope="col|row">`. A grid of `<div>` is not a table.

### ARIA — only when needed

ARIA is a fallback for cases where semantic HTML cannot express the relationship. The default rule is "use semantic HTML; reach for ARIA only when it is genuinely required". Common ARIA pitfalls:

- **Redundant roles** (`<button role="button">` is redundant; the semantic element already carries the role). Remove.
- **Wrong roles** (`role="list"` on a `<div>` instead of using `<ul>`). Use the semantic element.
- **Missing accessible names** on icon-only buttons (`<button><Icon /></button>` with no `aria-label`). Add `aria-label="Close"`.
- **Live regions misused** (every state change announced as `aria-live="assertive"`). Use `aria-live` sparingly — only for genuinely time-critical updates (e.g. a toast notification, a form submission result).

### Colour and contrast

- Text contrast ≥ **4.5:1** against background (large text ≥ 18pt / 14pt bold can be 3:1).
- Non-text contrast ≥ **3:1** for UI affordances (focus rings, borders, icons that convey state).
- Do not rely on colour alone to convey information (a red label without an icon / text means a colour-blind user misses the signal).

### Motion and timing

- Respect `prefers-reduced-motion` — disable non-essential animations when the user has it set.
- Timeouts (e.g. session expiry) provide a warning and a way to extend; do not silently log the user out.
- Auto-playing content has pause / stop / mute controls within 3 seconds of start.

## Responsive design

Three principles, in order:

1. **Mobile-first.** Design the layout for 320-375px width first (one column, finger-sized targets, content prioritised top-down). Tablet (≥ 768px) and desktop (≥ 1024px) layouts add density / multi-column / sidebar, but never remove information.
2. **Fluid before fixed.** Use `max-width` / `min-width` / `clamp(...)` / `auto-fit` / `auto-fill` grid; avoid pinning a layout to a single pixel width. The viewport is a continuum, not three breakpoints.
3. **Touch targets ≥ 44×44px.** Buttons, links, icon-targets — anything tappable hits the WCAG minimum. Tightly-spaced touch targets (8px apart or less) are unreliable on mobile.

Breakpoints (defaults; override in the project's design-system tokens):

| breakpoint | min-width | typical layout |
| --- | --- | --- |
| mobile | 0 | single column, drawer nav |
| tablet | 768px | 2-column, persistent sidebar where useful |
| desktop | 1024px | 2-3 column, persistent sidebar |
| wide | 1440px | 3+ column or content-max-width clamp |

Test surfaces: 320px (narrow phone), 375px (iPhone SE), 768px (iPad portrait), 1024px (small laptop), 1440px (typical desktop). The Lighthouse mobile preset + Chrome DevTools device emulation at these widths catches most regressions.

## Common rationalizations

| rationalization | truth |
| --- | --- |
| "The component library doesn't have this exact variant — I'll just style it inline." | Local extension is design-system entropy. Propose the variant to the system as a follow-up slug; in the meantime, the existing variant + a `Notes:` line ("AC-3 ships with `Button variant=secondary`; design-system PR to add `destructive` filed as #N") is the audit-trailed fix. |
| "It's just a prototype — accessibility can come later." | Prototypes ship. The "real" version that fixes accessibility almost always inherits the prototype's component tree, which encoded the inaccessible defaults. Cheaper to be accessible from line one. |
| "I'll use `aria-label` to make this `<div onClick>` a button." | A `<div>` is not a button. Screen readers, voice control, keyboard navigation, and the OS-level focus contract all treat `<button>` differently. `aria-label` paints over the surface; the semantic gap remains. Use `<button>`. |
| "Centering looks cleaner — every form should be centered." | Centered forms with left-aligned labels are the AI default; left-aligned forms read faster (eye returns to the left margin on each new line) and respect the reading direction. Use centered layout for hero / empty-state / modal-picker only. |
| "The user said `make it modern` — purple gradient + rounded-2xl is modern." | "Modern" is a brief, not a checklist of stylistic clichés. Translate to specifics: which spacing scale, which radius scale, which type scale, which colour palette. Surface a clarifying question if the user's "modern" is undefined; do not autogenerate the AI-aesthetic default. |
| "Adding a `transition-all` is harmless — better to animate than not." | `transition-all` animates every property change, including layout-shifting ones (`padding`, `margin`, `gap`). The result is jank: the page jiggles on every state change. Animate specific properties (`opacity`, `transform`) that don't trigger layout. |
| "I'll skip the focus indicator — the design looks cleaner without it." | Removing focus indicators makes the UI unusable for keyboard users (15-20% of users have some keyboard-dominant workflow). It is a `required` accessibility violation. Style the focus indicator to match the design; do not remove it. |
| "Everyone uses Lucide icons — adding them is a quick polish." | Decorative icons add cognitive load. Each icon a user has to interpret is a tax on reading. Use icons that add affordance (next, expand, complete) or carry information (status, type); skip the decorative sprinkle. |

## Red flags

Stop and re-read this skill if any appear in the diff:

- **A new colour value not from the design system token list** (`#7c3aed`, `rgb(...)`, `hsl(...)` inline). Token or no merge.
- **A new spacing value not from the design system scale** (`padding: 23px`, `margin-top: 14px`). Token or no merge.
- **A `<div>` with `onClick` and no `role` / `tabindex` / keyboard handler**. Replace with `<button>` or `<a>`.
- **A form element without an associated `<label>`**. `aria-label` on the input is a workaround; the proper label is the win.
- **`outline: none` without an equivalent replacement focus indicator**. The page is no longer keyboard-accessible.
- **A heading skip** (`<h1>` directly to `<h3>`). The page structure is malformed for screen readers.
- **A grid of `<div>` masquerading as a table.** Use `<table>` semantics when the data is tabular.
- **An animation with `duration > 300ms` on a state change** (e.g. a modal open transition that takes 600ms). Slow animations make the UI feel laggy and trip `prefers-reduced-motion`.
- **A layout that breaks at < 375px viewport width.** Test at 320px; fix before merging.
- **A button with > 50 chars of text.** The button's action is unclear; either rename or replace with a link + helper text.
- **Mixing two breakpoint conventions** (e.g. one component uses `md:flex` and another uses `@media (min-width: 800px)`). Pick one; the system's choice wins.

## Verification

Before committing UI work, before the GREEN-phase commit:

- [ ] Every component renders correctly at 320px / 375px / 768px / 1024px / 1440px (test via Chrome DevTools device emulation or a Storybook viewport addon).
- [ ] Every interactive element is reachable by keyboard (tab through the page; nothing is skipped).
- [ ] Every interactive element has a visible focus indicator with ≥ 3:1 contrast.
- [ ] Text contrast ≥ 4.5:1 (large text ≥ 3:1) — check with a contrast-ratio tool, not by eye.
- [ ] Colours / spacing / typography / radius / shadow use design-system tokens; no inline hex / inline px / inline rem unless documented as a deviation.
- [ ] Components use semantic HTML (`<button>` / `<a>` / `<ul>` / `<table>` / `<label>`) — no `<div role="button">`.
- [ ] ARIA only where semantic HTML cannot express the relationship; no redundant roles.
- [ ] Animations respect `prefers-reduced-motion`.
- [ ] AI-aesthetic anti-pattern table walked; deviations recorded as `Notes:`.
- [ ] Component-architecture rules walked: composition over configuration (no 7-boolean props); controlled-vs-uncontrolled is one per component.

If any box is unchecked → fix or surface as a deviation with explicit `Notes:` text. UI quality is part of the AC's `verifiable` line; this skill body is what the `verifiable` line cites.
